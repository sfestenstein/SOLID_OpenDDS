## Plan: Extract `PubSubOpenDds` service facade (Iteration 1)

Evolve the inline-OpenDDS MVP into a small SOLID library named **`pub_sub_open_dds`** that hides OpenDDS behind a typed service facade with a `pre_activate / activate / post_activate / deactivate` lifecycle. This first iteration only extracts the facade, adds typed publisher/subscriber handles via a `DdsTraits<T>` specialization, ports the existing programs to consume the library as examples, and stands up a GoogleTest suite. No lambda registry, no QoS API, no docs site yet.

**Steps**

**Phase A — Library scaffolding** *(blocks everything else)*
1. Create `include/pub_sub_open_dds/`, `src/pub_sub_open_dds/`, `examples/publisher`, `examples/subscriber`, `tests/`.
2. In CMakeLists.txt, add static-library target `pub_sub_open_dds` linked publicly to `OpenDDS::Dcps`/`Rtps`/`Rtps_Udp`, plus `PUBSUB_OPEN_DDS_BUILD_EXAMPLES` and `PUBSUB_OPEN_DDS_BUILD_TESTS` options.

**Phase B — Core abstractions** *(parallel sub-steps)*
3. `include/pub_sub_open_dds/dds_traits.h` — primary template `DdsTraits<T>` documenting the required member types (TypeSupport/Impl/Var, DataWriter, DataReader, plus `_var` aliases).
4. `include/pub_sub_open_dds/config.h` — `ServiceConfig { domain_id, config_file, argc/argv }`.
5. `include/pub_sub_open_dds/lifecycle_state.h` — enum used to enforce call ordering.

**Phase C — Service facade** *(depends on B)*
6. `include/pub_sub_open_dds/service.h` + `src/pub_sub_open_dds/service.cpp` — non-copyable `Service` owning the factory/participant; `pre_activate`, templated `register_publisher<T>` / `register_subscriber<T>` (record + type-erased realize thunk), `activate` (registers types, creates topics dedup'd by name, builds the writers/readers with today's reliable QoS), `post_activate`, idempotent `deactivate` that calls `TheServiceParticipant->shutdown()`.

**Phase D — Typed handles** *(depends on C)*
7. `include/pub_sub_open_dds/publisher.h` — header-only `Publisher<T>` with `write(const T&)` and `wait_for_subscribers(timeout, min_count=1)` reusing today's `StatusCondition`+`WaitSet` loop from publisher.cpp.

**Phase E — Port apps to examples** *(depends on D)*

// Notional radar-system IDL. Five topic types exchanged between SensorApp
// and WorkstationApp. Built with the OpenDDS IDL-to-C++11 mapping.
module RadarSystem {

  enum ComponentState {
    COMPONENT_OFFLINE, COMPONENT_INITIALIZING, COMPONENT_ONLINE,
    COMPONENT_DEGRADED, COMPONENT_FAULT
  };

  @topic
  struct ComponentStatus {
    string sensor_id;
    @key string component_id;
    ComponentState state;
    string message;
    unsigned long long timestamp_ns;
  };

  @topic
  struct RadarTrack {
    string sensor_id;
    @key unsigned long track_id;
    double range_m;
    double bearing_deg;
    double elevation_deg;
    double speed_mps;
    unsigned long long timestamp_ns;
  };

  enum CommandType {
    CMD_START_SCAN, CMD_STOP_SCAN, CMD_SHUTDOWN, CMD_RESET, CMD_SET_SCAN_RATE
  };

  @topic
  struct Command {
    string target_sensor_id;
    @key unsigned long command_id;
    CommandType type;
    string parameters;
    unsigned long long timestamp_ns;
  };

  enum CommandResult {
    CMD_ACCEPTED, CMD_REJECTED, CMD_COMPLETED, CMD_FAILED
  };

  @topic
  struct CommandStatus {
    string sensor_id;
    @key unsigned long command_id;
    CommandResult result;
    string message;
    unsigned long long timestamp_ns;
  };

  @topic
  struct ComponentStatusRequest {
    string target_sensor_id;
    @key unsigned long request_id;
    string component_id_filter;   // empty = all components
    unsigned long long timestamp_ns;
  };
};

// SensorApp — notional radar sensor on the pub_sub_open_dds facade.
// Publishes ComponentStatus + RadarTrack periodically and CommandStatus on demand.
// Subscribes to Command and ComponentStatusRequest from a workstation.
#include "radar_traits.h"
#include "pub_sub_open_dds/service.h"

#include <atomic>
#include <chrono>
#include <csignal>
#include <cstdint>
#include <iostream>
#include <mutex>
#include <string>
#include <thread>

namespace {
constexpr DDS::DomainId_t DOMAIN_ID = 43;
const std::string         SENSOR_ID = "sensor_alpha";

std::atomic<bool> g_running{true};
std::mutex        g_cout_mtx;
void handle_sigint(int) { g_running.store(false); }

uint64_t now_ns() {
  return std::chrono::duration_cast<std::chrono::nanoseconds>(
      std::chrono::system_clock::now().time_since_epoch()).count();
}

const char* to_string(RadarSystem::ComponentState s) {
  switch (s) {
    case RadarSystem::ComponentState::COMPONENT_OFFLINE:      return "OFFLINE";
    case RadarSystem::ComponentState::COMPONENT_INITIALIZING: return "INIT";
    case RadarSystem::ComponentState::COMPONENT_ONLINE:       return "ONLINE";
    case RadarSystem::ComponentState::COMPONENT_DEGRADED:     return "DEGRADED";
    case RadarSystem::ComponentState::COMPONENT_FAULT:        return "FAULT";
  }
  return "?";
}
const char* to_string(RadarSystem::CommandType t) {
  switch (t) {
    case RadarSystem::CommandType::CMD_START_SCAN:    return "START_SCAN";
    case RadarSystem::CommandType::CMD_STOP_SCAN:     return "STOP_SCAN";
    case RadarSystem::CommandType::CMD_SHUTDOWN:      return "SHUTDOWN";
    case RadarSystem::CommandType::CMD_RESET:         return "RESET";
    case RadarSystem::CommandType::CMD_SET_SCAN_RATE: return "SET_SCAN_RATE";
  }
  return "?";
}

void publish_status(pub_sub_open_dds::Publisher<RadarSystem::ComponentStatus>& pub,
                    const std::string& component_id,
                    RadarSystem::ComponentState state,
                    const std::string& msg) {
  RadarSystem::ComponentStatus s;
  s.sensor_id(SENSOR_ID);
  s.component_id(component_id);
  s.state(state);
  s.message(msg);
  s.timestamp_ns(now_ns());
  pub.write(s);
}
} // namespace

int main(int argc, char* argv[]) {
  using namespace pub_sub_open_dds;
  std::signal(SIGINT, handle_sigint);
  try {
    Service svc;
    ServiceConfig cfg; cfg.domain_id = DOMAIN_ID; cfg.argc = argc; cfg.argv = argv;
    svc.pre_activate(cfg);

    auto status_pub     = svc.register_publisher<RadarSystem::ComponentStatus>("ComponentStatus");
    auto track_pub      = svc.register_publisher<RadarSystem::RadarTrack>("RadarTrack");
    auto cmd_status_pub = svc.register_publisher<RadarSystem::CommandStatus>("CommandStatus");

    svc.register_subscriber<RadarSystem::Command>(
        "Command",
        [cmd_status_pub](const RadarSystem::Command& cmd) {
          if (!cmd.target_sensor_id().empty() && cmd.target_sensor_id() != SENSOR_ID) return;
          { std::lock_guard<std::mutex> lk(g_cout_mtx);
            std::cout << "[sensor] command #" << cmd.command_id()
                      << " type=" << to_string(cmd.type())
                      << " params='" << cmd.parameters() << "'\n"; }
          RadarSystem::CommandStatus cs;
          cs.sensor_id(SENSOR_ID);
          cs.command_id(cmd.command_id());
          cs.result(RadarSystem::CommandResult::CMD_ACCEPTED);
          cs.message("acknowledged");
          cs.timestamp_ns(now_ns());
          cmd_status_pub->write(cs);
        });

    svc.register_subscriber<RadarSystem::ComponentStatusRequest>(
        "ComponentStatusRequest",
        [status_pub](const RadarSystem::ComponentStatusRequest& req) {
          if (!req.target_sensor_id().empty() && req.target_sensor_id() != SENSOR_ID) return;
          { std::lock_guard<std::mutex> lk(g_cout_mtx);
            std::cout << "[sensor] status request #" << req.request_id()
                      << " filter='" << req.component_id_filter() << "'\n"; }
          publish_status(*status_pub, "alpha.radar_array",
                         RadarSystem::ComponentState::COMPONENT_ONLINE, "on-demand report");
          publish_status(*status_pub, "alpha.signal_processor",
                         RadarSystem::ComponentState::COMPONENT_ONLINE, "on-demand report");
        });

    svc.activate();
    svc.post_activate();

    { std::lock_guard<std::mutex> lk(g_cout_mtx);
      std::cout << "[sensor] '" << SENSOR_ID << "' up on domain " << DOMAIN_ID
                << " (Ctrl-C to stop)\n"; }

    uint32_t tick = 0;
    while (g_running.load()) {
      publish_status(*status_pub, "alpha.radar_array",
                     RadarSystem::ComponentState::COMPONENT_ONLINE, "heartbeat");
      publish_status(*status_pub, "alpha.signal_processor",
                     RadarSystem::ComponentState::COMPONENT_ONLINE, "heartbeat");
      for (uint32_t k = 0; k < 2; ++k) {
        RadarSystem::RadarTrack t;
        t.sensor_id(SENSOR_ID);
        t.track_id(100u + k);
        t.range_m(5000.0 + (tick * 50.0) + k * 200.0);
        t.bearing_deg(static_cast<double>((tick * 5 + k * 90) % 360));
        t.elevation_deg(2.5 + k);
        t.speed_mps(150.0 + k * 25.0);
        t.timestamp_ns(now_ns());
        track_pub->write(t);
      }
      ++tick;
      for (int i = 0; i < 10 && g_running.load(); ++i)
        std::this_thread::sleep_for(std::chrono::milliseconds(100));
    }
    svc.deactivate();
    std::cout << "[sensor] shutdown clean\n";
  } catch (const std::exception& e) {
    std::cerr << "[sensor] " << e.what() << "\n";
    return 1;
  }
  return 0;
}

## Design notes
- **Domain 43**, distinct from the Messenger demo (domain 42), so the two demos don't accidentally cross-talk.
- **5 topics, one per type**, with the same name as the IDL struct (e.g. `"ComponentStatus"`). The facade dedups topics by name + type; using distinct names avoids the "same topic, different type" failure path.
- **Target addressing** is a plain `target_sensor_id` string on every workstation→sensor message. Empty string = broadcast. The SensorApp early-returns on mismatch — works fine for one workstation/many sensors and many workstations/one sensor.
- **Reply from callback**: the SensorApp's Command listener captures `cmd_status_pub` by value (shared_ptr) and calls `write()` from the DDS receive thread. OpenDDS writers are thread-safe for `write()`, so this is fine.
- **Shutdown**: a SIGINT handler flips an atomic; main loops in 100 ms slices so Ctrl-C is responsive; `Service::deactivate()` runs explicitly (and again via the destructor as a safety net).

## Suggestions you asked for

**More message types worth adding** (each exercises a specific QoS shape):
1. **`Alert` / `SystemAlarm`** — sensor → workstation, *high-priority* events (radar fault, power dropout, RF interference). Natural fit for **`latched`** (REL + transient_local + KEEP_LAST 1 per `@key alarm_id`): every alarm "stays on the wall" until cleared, and a late workstation sees the current set instantly. Could add a `severity` enum + `acknowledged_by` field.
2. **`OperatorAuditLog`** — operator action history (every `Command` issued, every config change). Perfect for **`event_bus`** (REL + volatile + KEEP_ALL): never drop a record, but only callers present during the session need to see them. Useful baseline for a future *persistent* profile.
3. **`RawIQSample` / `BeamTelemetry`** — high-rate raw signal samples, ~kHz. **`streaming`** (BE + KEEP_LAST 1): newest sample wins, drops under load are fine. This is the inverse extreme of `reliable_transient` and would visibly drop frames if you ran it next to a slow subscriber.
4. **`TrackingCue`** — workstation → sensor, "look here". Reasonable place to try **ownership / exclusive ownership** in a future iteration: multiple workstations may issue cues but one wins, deterministically.
5. **`OperatorChat`** — workstation ↔ workstation. Reasonable fit for **`reliable`** with a deeper history (KEEP_LAST 50). Mostly there to demonstrate the demo isn't strictly hierarchical.

**Profiles worth adding** when you grow the QoS surface area:
- **`Persistent`** — REL + TRANSIENT_LOCAL + KEEP_ALL, paired with `durability_service.history_depth` so even crash-restart of the writer doesn't lose history (needs OpenDDS's persistent durability provider config — heavier).
- **`Heartbeat`** — BE + KEEP_LAST 1 *plus* a `deadline` QoS the reader monitors. Lets the workstation actually detect a stale sensor instead of silently believing the last value.
- **`Critical`** — REL + KEEP_ALL with bounded `resource_limits` and a `liveliness` QoS at `MANUAL_BY_TOPIC`; failing to assert liveliness drops the writer from the matched set so subscribers see the loss.
- **Partitions** — not a profile, but a knob the `ServiceConfig` should grow: lets you isolate the radar demo's `Command` traffic per workstation without separate domains.

## Summary

**QoS profile facility** (qos_profile.h, qos_profile.cpp)
- Extended `QosProfile` with four new optional knobs (default 0/false == "leave at OpenDDS default"): `deadline_ms`, `liveliness_lease_ms`, `liveliness_manual`, `max_samples`, `max_instances`. Both `apply()` overloads honour them.
- Three new built-in profiles:
  - **`persistent`** — REL + TRANSIENT_LOCAL + KEEP_ALL. A service-local stand-in for true DDS PERSISTENT (no on-disk store yet; intent is documented so a future switch to real PERSISTENT only changes the struct).
  - **`heartbeat`** — BE + KEEP_LAST 1 + 3 s DEADLINE. A silent writer triggers `on_requested_deadline_missed` on the reader instead of going unnoticed.
  - **`critical`** — REL + KEEP_ALL + MANUAL_BY_TOPIC liveliness (5 s lease) + bounded `max_samples=1000`. Writer must actively assert liveliness or drop from the matched set.

**Five new IDL message types** (idl)
- SystemAlarm.idl — `@key alarm_id`, `AlarmSeverity` enum, `acknowledged_by`.
- OperatorAuditLog.idl — `AuditAction` enum, free-form `detail`.
- RawIQSample.idl — `seq_no`, `center_freq_hz`, paired bounded `IQVector` sequences.
- TrackingCue.idl — `ownership_strength` for a future EXCLUSIVE_OWNERSHIP experiment.
- OperatorChat.idl — workstation-to-workstation chat.

    /**
     * init
     *
     * qos_profile profile should be formatted like:
     *
     *    qos_base_file_name_without_extension#profile_name_in_xml_file
     *
     * Init parses this string and will append ".xml" to
     * qos_base_file_name_without_extension. It'll than invoke
     * the init method on the XML_File_Intf class.
     *
     */
    DDS::ReturnCode_t
    init (const ACE_TCHAR * qos_profile);

    //@{
    /**
     *
     * These methods have DDS QOS, a profile and a topic_name as
     * input.
     *
     * The DDS QOS is passed by reference. This library fills this
     * QOS and will return it to the callee.
     *
     * qos_profile should be the same profile string as the one
     * passed to the init method.
     *
     * If the profile name is correct, the methods will invoke
     * the corresponding method in the XML_File_Intf class.
     *
     */
    DDS::ReturnCode_t
    get_datawriter_qos (DDS::DataWriterQos& dw_qos,
                        const ACE_TCHAR *qos_profile,
                        const ACE_TCHAR* topic_name);

    get_datareader_qos (DDS::DataReaderQos& dr_qos,

                        const ACE_TCHAR* topic_name);

    get_publisher_qos (DDS::PublisherQos& pub_qos,

## What's bloating the package

The current shape is **11 headers + 5 .cpp files**:

| File pair | Lines | What it actually contains |
|---|---|---|
| error.h | 15 | one `Error` exception class |
| lifecycle_state.h | 15 | one enum |
| config.h | 18 | one `ServiceConfig` struct |
| `qos_applier.h/.cpp` | 28 + 15 | a 3-field struct + a 5-line factory |
| `qos_xml_loader.h/.cpp` | 50 + 100 | thin wrapper over `OpenDDS::DCPS::QOS_XML_Loader` |
| `topic_config.h/.cpp` | 70 + 120 | INI parser + binding lookup |
| `qos_profile.h/.cpp` | 90 + 170 | profile struct + 9 named built-ins + `apply()` |
| `service.h/.cpp` | 175 + 130 | the actual facade |
| `publisher.h`, `subscriber.h`, `dds_traits.h` | ~90 each | the typed handles + traits |

Three of those headers are one-type files that pull their weight only because they pre-date the consolidation phase. And the QoS module has **three** layers (`QosProfile` → `QosApplier` → `QosXmlLoader`) where two of them earn their keep only because the third exists.

## Where I'd cut, and why

### 1. Let `TopicConfig` own the OpenDDS XML loader directly → delete `qos_xml_loader.h/.cpp`

You're right that in mature DDS stacks the **DDS layer is the XML parser**. OpenDDS already gives us `OpenDDS::DCPS::QOS_XML_Loader`. Our `QosXmlLoader` wrapper currently adds:
- a `shared_ptr<QosXmlLoader>` facade so applier closures stay alive;
- string-trim of `.xml`;
- exception translation on parse failure;
- formatted `xml:base#profile` log strings.

All of that is ~15 lines of real work and belongs *inside* `TopicConfig`, which is already the runtime QoS entry point. The user flow becomes:

```cpp
auto cfg = TopicConfig::load_from_file("workstation_topics.ini");
cfg.use_xml_qos_file("radar_qos.xml");   // calls OpenDDS::QOS_XML_Loader::init internally
svc.register_publisher<T>("RadarTrack", cfg.qos_for("RadarTrack"));
```

That kills the `QosXmlLoader` indirection entirely — one less concept, one less header pair. **TopicConfig becomes the single source of truth for "topic name → QoS"**, which is what your INI-style mental model already implies.

### 2. Fold `QosApplier` into topic_config.h and rename → delete `qos_applier.h/.cpp`

`QosApplier` is just a pair of `std::function`s with a name string for logging. It's not a stand-alone concept — it's an *internal* detail of how TopicConfig hands a QoS to Service. Two options:

- **Simpler**: move the `QosApplier` struct definition into topic_config.h (it's literally where it belongs — TopicConfig produces them, Service consumes them) and drop the standalone header. `Service::register_*` keeps its templated `QosApplier` overload but the type now travels with topic_config.h.
- **Even simpler**: drop the struct entirely; have `TopicConfig::qos_for("topic")` return a small concrete `QosBinding` value that knows how to stamp both writer and reader QoS, where the implementation is either "apply this `QosProfile`" or "ask the XML loader for these (profile, topic) names". Same surface, less type-erasure machinery.

I'd go with the first — it preserves the existing API exactly and just deletes one file.

### 3. Inline error.h, lifecycle_state.h, config.h into service.h

All three are tiny single-type headers used **exclusively** by Service (`ServiceConfig` is only passed to `pre_activate`; `LifecycleState` is only the return type of `Service::state()`; `Error` is thrown from Service and TopicConfig/QosXmlLoader — that last one is the only mild argument against). The cost of three separate headers is more `#include` overhead than insight.

Inlining drops three files with zero behavioural change. If you want a *softer* version, fold error.h + lifecycle_state.h into a single `common.h` and keep config.h (since `ServiceConfig` is a documented user-facing struct).

### 4. Drop the `TopicConfig::qos_for(topic, default)` backward-compat shim

It exists to support a previous-iteration API that no caller uses anymore. The header even says "Prefer applier_for() for new code." A package this small doesn't need deprecation shims.

### Result

11 headers + 5 .cpp → **7 headers + 3 .cpp**, with a clearer mental model:

- service.h (absorbs config/error/lifecycle_state) + `service.cpp`
- `publisher.h`, `subscriber.h`, `dds_traits.h` — typed handles
- qos_profile.h (built-in profile catalog + `apply()`) + qos_profile.cpp
- topic_config.h (the *only* runtime QoS entry point: INI bindings + optional XML pass-through, owns `OpenDDS::DCPS::QOS_XML_Loader`) + topic_config.cpp

The user-facing story becomes: "use a `QosProfile` directly for trivial cases; use a `TopicConfig` (with an optional XML file) for everything else." No `QosApplier` to learn, no `QosXmlLoader` to construct.

### What I'd deliberately *not* touch

- **Keep `QosProfile` and the `qos::` built-in catalog.** The smoke test, examples, and "I just want reliable" cases all benefit from a programmatic shorthand that doesn't require shipping an XML file. Real DDS systems (RTI, Cyclone) all support both modes.
- **Keep `dds_traits.h` separate** — that's the user's specialisation point and deserves its own file.
- **Keep `publisher.h` and `subscriber.h` separate** — they're substantial (each carries its own listener / WaitSet helpers), and merging them muddies the typed-handle story.

## Quick preview of the design tension you'll be navigating (worth thinking about beforehand)

Three things drive most of the design choices here, and they pull against each other:

1. **Zero OpenDDS in user headers** — means hiding `DDS::ReturnCode_t`, `DataWriterQos`, the `*TypeSupportImpl` includes behind a PIMPL boundary or a code-generation step. PIMPL is cleaner on the user side but kills the template-based typed handles; code-gen is more invasive on the build.
2. **Type-safe typed handles** — means `Publisher<MyType>::write(const MyType&)` has to know `MyType`'s OpenDDS narrow at instantiation time. The only way to keep this *and* hide OpenDDS is to put `MyType`-specific glue (the trait) in a *generated* file that the user doesn't author.
3. **Testability** — fakes need to substitute for the OpenDDS runtime. That's much easier if `Service` talks to an `IDdsRuntime` interface, not to OpenDDS directly. Same idea for each `Publisher`/`Subscriber`.

There are clean ways to satisfy all three — typically: one user-authored IDL file → CMake helper generates both the OpenDDS trait *and* a typed `IPublisher<T>` / `ISubscriber<T>` user-facing pair; the facade lives behind an `IPubSubRuntime` interface with an OpenDDS implementation and a fake for tests. ## 3. QoS facility (iterations 2–5, current shape)

- `qos_profile.h` exposes a `QosProfile { name, reliable, durable, keep_all,
  history_depth, deadline_ms, liveliness_lease_ms, liveliness_manual,
  max_samples, max_instances }` struct + `apply()` overloads for both
  `DataWriterQos` and `DataReaderQos`. Built-in named profiles in
  `pub_sub_open_dds::qos`: `best_effort`, `reliable`, `reliable_transient`,
  `event_bus`, `latched`, `streaming`, `persistent`, `heartbeat`,
  `critical`. `find_builtin_profile(name)` does a case-insensitive lookup.
- `QosApplier { writer, reader, name }` (defined in `service.h`) is the
  type-erased value `Service` consumes. `make_applier(QosProfile)` adapts a
  built-in or hand-built profile.
- **There is no separate `qos_applier.h` / `qos_xml_loader.h` any more**:
  the XML path lives entirely inside `TopicConfig::use_xml_qos_file(path)`,
  which holds the `OpenDDS::DCPS::QOS_XML_Loader` as a `unique_ptr` to a
  forward-declared type.
- `TopicConfig::load_from_file(ini_path)` + optional
  `use_xml_qos_file(xml_path)` is the **single runtime QoS entry point**.
  `applier_for(topic, default)` resolves built-in names directly,
  dispatches `xml:<profile>` references through the loader, and falls back
  to `default` with a `std::cerr` warning on any miss.
- `Service::register_publisher<T>` / `register_subscriber<T>` accept either
  a `QosApplier` (from `TopicConfig::applier_for`) or a `QosProfile`
  (forwarded through `make_applier`). The default is `qos::reliable()` to
  preserve iteration-1 behaviour.

## 4. OpenDDS XML QoS gotchas

1. The CMake target name imported by `find_package(OpenDDS)` is
   `OpenDDS::QOS_XML_XSC_Handler` — **not**
   `OpenDDS::OpenDDS_QOS_XML_XSC_Handler`. The helper macro strips the
   `OpenDDS_` prefix when it builds the imported target name.
2. The schema `/opt/OpenDDS/docs/schema/dds_qos.xsd` restricts
   `qos_profile name=` to `[a-zA-Z0-9 ]+` — **no underscores**. Use
   `RadarTrackStreaming`, not `RadarTrack_Streaming`. Validation failure
   surfaces as a vanilla `rc=1` from `QOS_XML_Loader::init`; running
   `xmllint --noout --schema dds_qos.xsd <file>.xml` gives the precise
   reason.
3. OpenDDS resolves the schema via the `DDS_ROOT` env var
   (`$DDS_ROOT/docs/schema/`), set by `/opt/OpenDDS/setenv.sh`. The
   dev-container sources this in `/etc/bash.bashrc`, but non-interactive
   processes (e.g. CI) need to source it explicitly or the XML loader will
   silently fail.

## 5. OpenDDS IDL CMake gotchas (learned the hard way)

1. **Don't list IDL files from a parent CMakeLists when they live in a
   subdir.** `OPENDDS_TARGET_SOURCES` auto-adds
   `-I${CMAKE_CURRENT_SOURCE_DIR}` to opendds_idl. If your IDL files sit
   in a subdir and you also explicitly add `-I${SRC}/sub`, opendds_idl
   resolves cross-IDL `#include`s through BOTH paths and emits duplicate
   `#include`s in the generated headers (e.g. both `#include "CommonC.h"`
   AND `#include "sub/CommonC.h"`), and one of them won't exist.
   **Fix**: give the IDL files their OWN `CMakeLists.txt` in their dir so
   `CMAKE_CURRENT_SOURCE_DIR` *is* that dir; sibling
   `#include "Common.idl"` then resolves through the single auto-added
   path.
2. **List IDL files PRIVATE, not PUBLIC, in `OPENDDS_TARGET_SOURCES`**
   when the IDL target lives in a different CMake directory from its
   consumers. PUBLIC propagates the generated `.idl` source files into
   every consumer's source list, but their `GENERATED` property was set
   in the IDL dir scope and isn't visible to the consumer's dir →
   "Cannot find source file". The public include path that exports the
   generated `.h` files is set separately by the helper and stays
   exported either way.
3. **`sequence` is a reserved IDL keyword.** A `@topic` struct field
   named `sequence` makes opendds_idl emit a bare "syntax error". Use any
   other identifier (the demo uses `seq_no`).
4. **Fixed-size arrays via `typedef T name[N]` are rejected by
   opendds_idl** when used as struct fields under the C++11 mapping (also
   surfaces as "Illegal syntax or missing identifier following member
   type"). Use `typedef sequence<T, N> name;` instead — generates a
   bounded `std::vector<T>` under the C++11 mapping.

## 6. Critical facade invariant — don't break this

`Service` **MUST** keep registered `Publisher<T>` / `Subscriber<T>` handles
alive for its own lifetime. The OpenDDS listener installed by
`register_subscriber` holds a raw pointer back into
`Subscriber<T>::counter_`; if the caller discards the returned `shared_ptr`,
the handle dies as soon as `activate()` clears its pending-thunk capture and
the next sample triggers a use-after-free (heap corruption,
`malloc(): unaligned tcache chunk`).

The fix landed in `Service` via `std::vector<std::shared_ptr<void>>
handle_keepalive_`, cleared at the end of `deactivate()` **after**
`delete_contained_entities()` has drained in-flight callbacks. Don't remove
that vector unless you've replaced the listener-pointer-back-into-handle
pattern with something safer.

## 7. Conventions verified

- IDL uses the **C++11 mapping**:
  `OPENDDS_TARGET_SOURCES(... OPENDDS_IDL_OPTIONS -Lc++11)`. Generated
  members are `std::string` / `int32_t`; accessors are `m.from()` /
  `m.from("val")`, not `m.from = ...`.
- CMake options: `PUBSUB_OPEN_DDS_BUILD_EXAMPLES`,
  `PUBSUB_OPEN_DDS_BUILD_TESTS` (default ON).
- Facade lifecycle: `pre_activate → register_* → activate → post_activate
  → deactivate`. Out-of-order calls throw `pub_sub_open_dds::Error`.
- Hot paths return `DDS::ReturnCode_t` (deliberate; lifecycle/config
  throws). This is one of the documented OpenDDS leaks into user code.
- `Service` is non-copyable and non-movable; the destructor calls
  `deactivate()`.

## 8. Known intentional OpenDDS leaks into user code

These are the things the next architectural pass needs to address if the
goal is "zero OpenDDS in user headers":

1. `Publisher<T>::write` returns `DDS::ReturnCode_t`, so user code
   compares against `DDS::RETCODE_OK`. To fully hide DDS::, switch `write`
   to return `bool` or a custom enum.
2. `QosApplier::{writer,reader}` is `std::function<void(DDS::DataWriterQos&)>`
   / `std::function<void(DDS::DataReaderQos&)>` — a user that constructs
   a custom applier sees `DDS::DataWriterQos` directly. Hiding this needs
   a PIMPL-style opaque QoS handle.
3. The user-supplied `DdsTraits<T>` specialisation **must** include the
   OpenDDS-generated `<Type>TypeSupportImpl.h`, which transitively pulls
   in `dds/DCPS/*` everywhere it is included. Hiding this needs either
   PIMPL-erased typed handles or a CMake-driven code-generation step that
   emits both the trait *and* a user-facing typed pair.
4. `ServiceConfig` exposes `DDS::DomainId_t` and `ACE_TCHAR**` (the latter
   because `argv` is forwarded to `TheParticipantFactoryWithArgs`).

> The `pub_sub_open_dds` facade currently has 6 public headers, but the
> user-facing API (`Service`, `Publisher<T>`, `Subscriber<T>`, `QosApplier`,
> `QosProfile`, `TopicConfig`, `dds_traits`) still leaks OpenDDS types in
> several places: `DDS::ReturnCode_t` return values, `DDS::DataWriterQos` /
> `DataReaderQos` inside `QosApplier`, the user-supplied `DdsTraits<T>`
> specialisation requires the user to include the OpenDDS-generated
> `*TypeSupportImpl.h`, and `ServiceConfig` exposes `DDS::DomainId_t` +
> `ACE_TCHAR**`.
>
> I want a SOLID redesign that completely isolates OpenDDS from user
> code, and I want to talk about testability at the same time. Start by
> reading `docs/HANDOFF.md`, `README.md`, and the public headers under
> `pub_sub_open_dds/include/pub_sub_open_dds/`, then **propose where the
> seams should be** — don't write code yet.

### Three tensions to think about before that chat

1. **Zero OpenDDS in user headers** — means hiding `DDS::ReturnCode_t`,
   `DataWriterQos`, the `*TypeSupportImpl` includes behind a PIMPL
   boundary or a code-generation step. PIMPL is cleaner on the user side
   but kills the template-based typed handles; code-gen is more invasive
   on the build.
2. **Type-safe typed handles** — `Publisher<MyType>::write(const MyType&)`
   has to know `MyType`'s OpenDDS narrow at instantiation time. The only
   way to keep this *and* hide OpenDDS is to put `MyType`-specific glue
   (the trait) in a *generated* file the user doesn't author.
3. **Testability** — fakes need to substitute for the OpenDDS runtime.
   Much easier if `Service` talks to an `IDdsRuntime` interface, not to
   OpenDDS directly. Same idea for `Publisher` / `Subscriber`.

A clean way to satisfy all three: **one user-authored IDL file → CMake
helper generates both the OpenDDS trait and a typed `IPublisher<T>` /
`ISubscriber<T>` user-facing pair; the facade lives behind an
`IPubSubRuntime` interface with an OpenDDS implementation plus a fake for
tests.** Worth coming to the next chat with a position on this.

### Testing roadmap (layered, cheapest first)

- (a) Keep the integration smoke test; add more in-process round-trip
      tests covering each QoS profile. Realistic without any redesign.
- (b) Factor the runtime into an interface + add fakes so `Service` can
      be exercised without OpenDDS in unit-test scope. Only makes sense
      *after* the SOLID pass.
- (c) Full DI rewrite, GoogleTest harness, mocks for every collaborator.
      Overkill until (a) and (b) have shaken out the right interfaces.
